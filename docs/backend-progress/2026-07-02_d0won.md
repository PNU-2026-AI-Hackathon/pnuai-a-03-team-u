# 2026-07-02 작업 기록

## 구현한 것

- `app/ai/embeddings/openai_client.py` — OpenAI `text-embedding-3-small` 얇은 래퍼 (`embed_text`/`embed_texts`)
- `app/ai/embeddings/activity_embeddings.py` — embedding이 비어있는 Activity를 찾아 title+category+description으로 임베딩 생성, 100개씩 배치 처리
- `app/ai/recommendations/extracurricular_recommender.py` — 사용자 프로필(department + major + career_goal) 임베딩과 Activity 임베딩의 코사인 유사도 계산
  - `final_score = similarity_score * career_weight * recency_weight`
  - career_weight: career_goal 있으면 1.2배, 없으면 1.0배
  - recency_weight: 게시일 기준 90일 선형 감쇠 (최근 1.0 → 90일 0.5)
  - 마감 지난 활동/90일 이전 게시물은 후보에서 제외, 결과는 `UserActivityRecommendation`에 upsert
- `app/api/activities.py` — `GET /activities/recommendations/{user_id}` 추천 그리드 API
  - 캐시된 추천이 없으면 즉시 계산 후 반환 (최초 요청 시 지연 있음)
  - 응답에 d-day, 카테고리, 추천 점수(%) 포함
- `app/core/scheduler.py` — 자정 크롤 직후 임베딩 생성 → 중복 정리 → 전체 사용자 추천 재계산까지 이어서 실행하도록 확장
- `app/main.py` — activities 라우터 등록
- `requirements.txt` — `openai` 패키지 명시 추가 (기존엔 langchain-openai 통해 간접 설치됨)
- `app/ingestion/normalizers/dedup_activities.py` — 같은 출처 + 제목 완전 동일 + 게시일 3일 이내인 공지를 중복으로 간주해 조회수 높은 쪽만 남기고 삭제
  - 회차별/재모집 공고처럼 제목만 비슷하고 실제로 다른 공지는 게시일 간격 조건으로 보존
  - 자정 크롤 직후 자동 실행

## 고친 것

- `app/domains/activities/models.py`: `UserActivityRecommendation.user_id`/`activity_id`에 `ForeignKey(ondelete="CASCADE")` 추가, 마이그레이션(`ec32676909bf`) 생성 및 Supabase 적용
  - 기존엔 단순 정수 컬럼이라 유저/활동 삭제 시 추천 레코드가 고아로 남는 문제가 있었음
- `app/ingestion/crawlers/notice_board_crawler.py`: job 게시판 `fetch_job_board_page`에서 제목이 빈 문자열인 공지를 크롤링 단계에서 제외
  - 원인 조사 결과 크롤러 파싱 버그가 아니라, 원본 게시글(job 204025)이 텍스트 없이 이미지 배너만 올라온 공지였음. 제목을 알아낼 방법이 없고 추천에도 못 쓰므로 제외 처리, 기존 DB의 빈 제목 row도 삭제

## 막혔던 부분 / 삽질

1. 인증 시스템이 아직 없어서(로그인/JWT 미구현) 추천 API가 `user_id`를 경로 파라미터로 직접 받음 → 로그인 붙으면 `get_current_user` 의존성으로 교체 필요
2. job 게시판 빈 제목 공지 원인 조사 → 원본 게시글이 이미지 배너만 있고 텍스트 제목이 없는 케이스였음 (크롤러 버그 아님)
3. 제목 유사도 기반 중복 삭제를 검토하다가, 단순 80% 텍스트 유사도만으로는 회차별/재모집 공고(예: "하나은행" vs "한국은행" 채용설명회, "5월" vs "6월" 도서관 프로그램)까지 삭제될 위험을 발견 → 제목 완전 일치 + 같은 출처 + 게시일 3일 이내로 조건을 강화해 안전하게 처리
4. 마이그레이션 autogenerate가 FK 제약 이름을 `None`으로 생성해 downgrade 시 실패하는 문제 → 제약 이름을 명시적으로 지정
