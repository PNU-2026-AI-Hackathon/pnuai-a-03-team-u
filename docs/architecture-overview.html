Plan-U 아키텍처 개요

프론트엔드·백엔드·DB·API·배포까지 코드 실행 흐름과 함수 호출을 한 눈에.

1. 서비스 개요

Plan-U는 부산대(PNU) 재학생이 학업 정보를 입력하면 졸업요건을 분석하고 학년별 성장 로드맵과 다음 학기 시간표 후보를 추천하는 서비스다. 학생지원시스템(One-Stop)에서 성적표·학적·지도교수 정보를 자동 수집(Playwright 크롤링)해 판정에 활용하며, AI 상담 챗봇은 로드맵 편집을 제안 형태로 만들고 사용자가 승인한 것만 실제 반영된다(human-in-the-loop).

절대 원칙: 졸업요건 충족/미충족 판정은 규칙 기반 엔진만 한다. LLM은 판정 결과를 입력받아 설명·추천 문장만 만든다. 이 경계를 흐리는 변경은 하지 않는다.

개인정보(성적표 이미지, 이수내역, 자격증/어학 점수) 취급 원칙: 최소 수집, HTTPS 전송, 저장 시 암호화, OCR 원본은 텍스트 추출 직후 폐기, 외부 LLM 호출 시 개인 식별정보 비식별화.

6. 로드맵 상담 에이전트

구현: backend/app/domains/planning/roadmap_chat.py

설계 원칙

  • LangChain 단일 루프: LangGraph 같은 그래프 오케스트레이션은 쓰지 않는다. tool 호출 → 결과 반영 → 다음 tool 호출.
  • Human-in-the-loop: LLM이 로드맵을 직접 편집하지 않는다. propose_change가 pending_roadmap_changes에 제안만 쌓고, 사용자 승인 후 apply_pending_changes()에서 실제 반영.
  • tool_choice="any": LLM이 "일반 텍스트로 대충 답하고 끝"내는 탈출구를 없앤다. 사용자에게 보이는 메시지도 finish_response라는 도구로 나가야 함.
  • 규칙 기반 가드 우선: 학점 상한/과거 학기/이수 중복/계절수업 배치 등은 도구 계층에서 강제. LLM의 판단에만 맡기지 않는다.

Tool 카탈로그

Tool역할주요 리턴
get_graduation_progress주전공 이수구분별 남은 학점categories[], remaining_credits
get_roadmap_items현재 로드맵 + 학점 상한 + 이수기록items[], term_credit_cap, planned_credits_by_term, completed_courses, earliest_recorded_grade, current_academic_term, next_plannable_term
search_courses학생 학과·교육과정 스코프 안에서 과목 검색 (RAG)results[] (course_id, name, category, credits, grade, semester, description)
propose_changecreate/update/delete 제안. 실제 반영 아님change_id 또는 error (가드 위반 시)
finish_response사용자에게 보여줄 최종 답변message

가드 (propose_change에서 강제)

  1. 계절수업 vs 정규 학기: 여름계절수업 등 세션 전용 과목은 정규 1·2학기 슬롯에 배치 불가.
  2. 편입생 학년 하한: _min_completed_grade()가 반환한 값보다 낮은 planned_grade 거부. 편입 인정만 있는 상태(StudentCourseRecord.semester='입학전성적')에서도 3학년 폴백으로 판정.
  3. 과거 학기 create 차단: 현재 학기보다 이전으로는 새 항목 못 만듦.
  4. 학기당 학점 상한: 졸업기준학점 133+ → 21학점, 132- → 19학점. 초과 시 그 학기 기존 항목 목록과 대체 방향 hint를 함께 반환.
  5. 로드맵 내 중복: 같은 course_id가 이미 로드맵에 있거나 같은 세션에서 이미 제안됐으면 create 거부.
  6. 이수기록 중복: StudentCourseRecord의 이름과 정규화 매칭 (Ⅰ→I, 괄호·공백 제거). 매칭 시 create 거부.

시스템 프롬프트 컨텍스트 주입

매 대화 시작 시 _build_student_context_block(db, user)이 _SYSTEM_PROMPT에 학생 프로필을 붙인다:

  • 진로 목표 (user.career_goal)
  • 학적 프로그램: 주전공/복수전공/부전공/교직 각각의 학과·전공·교육과정 연도
  • 이수 완료 과목: 이수구분별 그룹핑

7. 졸업요건 판정 엔진

구현: backend/app/domains/academics/graduation_progress.py

현재 엔진은 이수구분별 총 학점만 대조하는 단순 엔진이다. 개별 과목 매칭(course_id) 로직은 아직 없다. StudentCourseRecord.category 합계 vs GraduationRequirement.required_* 필드값을 비교.

이 엔진은 규칙 기반이다. LLM은 이 결과를 입력받아 설명 문장만 생성한다. 판정 자체를 LLM이 하지 않는다는 원칙은 절대 지켜진다.

매핑: StudentCourseRecord.category(성적표 원문 정규화 값) → GraduationRequirement의 required_major_required, required_major_elective, required_general_required 등.

8. RAG 검색 (CurriculumRetriever)

구현: backend/app/ai/rag/curriculum_retriever.py

DB-first RAG. 벡터 검색은 옵션(use_vector=True)이고 기본은 structured DB 필터링이다. 학과/전공/교육과정연도 경계를 엄격히 지키기 위함.

필터 확장 규칙

  • grade='3' → Course.year IN ('3', '전학년')
  • semester='2학기' → Course.semester IN ('2', '1,2', '전학기')
  • category='교양필수' → alias 확장 → Course.category IN ('교양필수', '효원핵심교양', '기초교양')
  • category='교양선택' → Course.category IN ('교양선택', '효원균형교양', '효원창의교양')
  • 결과에서 (course_name, category) 기준 dedup — 학과별 다른 code로 시딩된 동일 개념 교양(공학작문및발표 등)을 한 번만 노출.

진로 키워드 확장 (career_keywords.py)

query 문자열에 "AI"/"보안"/"백엔드" 등 alias가 포함되면 관련 키워드를 확장해서 결과 랭킹에 사용. 다만 이 로직은 5개 카테고리만 있고 대부분의 학과는 커버되지 않는다.

9. 데이터 수집 (Ingestion)

Playwright 크롤러

backend/app/ingestion/crawlers/pnu_session.py가 세션 관리·SSO 로그인을 담당.

PNU SSO 특이사항: onestop.pusan.ac.kr에서 #global_login 링크를 클릭해서 login.pusan.ac.kr로 진입해야 csrfToken이 정상 렌더링된다. 직접 URL 진입 시 CSRF가 빈 문자열이라 로그인 후 세션 이관에 실패. 페이지 단위 재시도(_reach_login_form)와 실패 사유 진단 로깅(dialog/console/network)이 걸려 있음.

수집 항목

  • fetch_student_record: 학적부 (지도교수 이름은 API 응답 HB_ADVICE_PROF_NM에서 획득)
  • fetch_all_grades: 성적표 23건 예시
  • fetch_advisor_consultation: 상담 상태 (menuCD=449, 현재 학기 필터)

정규화 & 저장

pnu_normalizer.py가 성적표 raw → StudentCourseRecord로 매핑. 개별 course_id는 대부분 None(과목명만 저장)이라 이름 정규화 매칭에 의존.

11. Frontend 상태 관리

src/api/client.ts가 axios 인스턴스를 만들고, 요청 인터셉터에서 JWT를 Authorization: Bearer로 붙인다. 응답 인터셉터에서 401이면 AUTH_EXPIRED_EVENT를 발생시켜 자동 로그아웃 유도.

baseURL = import.meta.env.VITE_API_BASE_URL ?? "http://127.0.0.1:8000". 실제로는 .env에서 VITE_API_BASE_URL=http://127.0.0.1:8001로 지정.

22. Supabase (프로덕션 DB)

팀 공유 Postgres. backend/.env의 DATABASE_URL이 Supabase를 직접 가리킨다(별도 스테이징 없음).

주의: 마이그레이션이나 대량 데이터 반영처럼 되돌리기 어려운 작업은 로컬 Postgres에서 완전히 검증한 뒤 Supabase에 한 번에 반영. 절반만 반영된 상태를 만들지 않는다.

시드 스크립트는 대부분 (dept_id, major_id, course_code) 등 자연 키 기준 idempotent get-or-create라 재실행 안전. 그래도 --dry-run으로 예상 delta 먼저 확인.
